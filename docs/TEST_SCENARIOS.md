# 테스트 시나리오 문서

> 쇼핑몰 MVP의 전체 사용자 플로우를 테스트하기 위한 시나리오 문서

## 📋 테스트 시나리오 목록

### 시나리오 1: 정상 구매 플로우

**목표**: 사용자가 상품을 탐색하고 구매하는 전체 플로우를 검증합니다.

**단계**:
1. ✅ 홈페이지 접속 (`/`)
   - 히어로 섹션 표시 확인
   - 카테고리 그리드 표시 확인
   - 인기 상품 섹션 표시 확인

2. ✅ 회원가입/로그인
   - Clerk 로그인 페이지 접근
   - 테스트 계정으로 로그인
   - 로그인 후 홈페이지로 리다이렉트 확인

3. ✅ 상품 탐색
   - "전체 상품 보기" 버튼 클릭
   - 상품 목록 페이지 표시 확인
   - 카테고리 필터링 테스트
   - 정렬 기능 테스트 (최신순, 가격순 등)
   - 검색 기능 테스트
   - 페이지네이션 테스트

4. ✅ 상품 상세 페이지
   - 상품 카드 클릭
   - 상품 상세 정보 표시 확인
   - 재고 수량 표시 확인
   - 가격 표시 확인
   - "장바구니에 담기" 버튼 클릭

5. ✅ 장바구니 관리
   - 장바구니 페이지 접근 (`/cart`)
   - 추가한 상품 표시 확인
   - 수량 변경 테스트
   - 상품 삭제 테스트
   - 총액 계산 확인

6. ✅ 주문 생성
   - "주문하기" 버튼 클릭
   - 체크아웃 페이지 접근 (`/checkout`)
   - 배송지 정보 입력
   - 주문 메모 입력 (선택)
   - "주문하기" 버튼 클릭
   - 주문 생성 확인

7. ✅ 결제 진행
   - 결제 위젯 표시 확인
   - 결제 수단 선택 (카드, 계좌이체 등)
   - 테스트 결제 진행
   - 결제 성공 페이지 확인 (`/payment/success`)

8. ✅ 주문 내역 확인
   - 주문 상세 페이지 확인 (`/orders/[id]`)
   - 주문 상태 "결제 완료" 확인
   - 주문 상세 항목 확인
   - 주문 목록 페이지 확인 (`/orders`)
   - 주문 내역 표시 확인

**예상 결과**: 모든 단계가 정상적으로 완료되어야 합니다.

### 시나리오 2: 에러 케이스 테스트

**목표**: 예외 상황에서도 애플리케이션이 안정적으로 동작하는지 검증합니다.

#### 2.1 인증되지 않은 사용자 접근

**단계**:
1. 로그아웃 상태에서 장바구니 페이지 접근 (`/cart`)
2. 로그아웃 상태에서 주문 페이지 접근 (`/checkout`)
3. 로그아웃 상태에서 주문 내역 페이지 접근 (`/orders`)

**예상 결과**: 
- 로그인 페이지로 리다이렉트되어야 합니다.
- 에러 메시지가 표시되지 않아야 합니다.

#### 2.2 재고 부족 상품 주문 시도

**단계**:
1. 재고가 0인 상품의 상세 페이지 접근
2. "장바구니에 담기" 버튼 비활성화 확인
3. 재고가 부족한 상품을 장바구니에 추가 (직접 DB 수정)
4. 주문 생성 시도

**예상 결과**:
- 재고 부족 에러 메시지가 표시되어야 합니다.
- 주문이 생성되지 않아야 합니다.

#### 2.3 결제 취소

**단계**:
1. 주문 생성 후 결제 위젯 표시
2. 결제 수단 선택
3. 결제 창에서 취소 버튼 클릭

**예상 결과**:
- 결제가 취소되어야 합니다.
- 주문 상태가 "결제 대기 중"으로 유지되어야 합니다.
- 에러 메시지가 표시되지 않아야 합니다.

#### 2.4 결제 실패

**단계**:
1. 주문 생성 후 결제 위젯 표시
2. 결제 수단 선택
3. 결제 실패 시뮬레이션 (테스트 모드에서 실패 케이스)

**예상 결과**:
- 결제 실패 페이지로 리다이렉트되어야 합니다 (`/payment/fail`).
- 주문 상태가 "결제 실패"로 업데이트되어야 합니다.
- 에러 메시지가 표시되어야 합니다.

#### 2.5 빈 장바구니로 주문 시도

**단계**:
1. 장바구니가 비어있는 상태에서 주문 페이지 URL 직접 접근 (`/checkout`)

**예상 결과**:
- 장바구니 페이지로 리다이렉트되어야 합니다.
- 에러 메시지가 표시되지 않아야 합니다.

---

### 시나리오 3: 성능 테스트

**목표**: 애플리케이션의 성능을 검증합니다.

#### 3.1 페이지 로딩 속도

**단계**:
1. 각 주요 페이지의 로딩 시간 측정
   - 홈페이지 (`/`)
   - 상품 목록 (`/products`)
   - 상품 상세 (`/products/[id]`)
   - 장바구니 (`/cart`)
   - 주문 페이지 (`/checkout`)
   - 주문 내역 (`/orders`)

**예상 결과**:
- 각 페이지가 3초 이내에 로드되어야 합니다.
- 로딩 스켈레톤이 표시되어야 합니다.

#### 3.2 이미지 최적화

**단계**:
1. 상품 이미지가 Next.js Image 컴포넌트를 사용하는지 확인
2. 이미지 로딩 실패 시 플레이스홀더 표시 확인

**예상 결과**:
- 이미지가 최적화되어 로드되어야 합니다.
- 이미지 로딩 실패 시 기본 이미지가 표시되어야 합니다.

#### 3.3 대량 데이터 처리

**단계**:
1. 많은 수의 상품이 있을 때 상품 목록 페이지 성능 확인
2. 많은 수의 주문이 있을 때 주문 내역 페이지 성능 확인

**예상 결과**:
- 페이지네이션이 정상적으로 작동해야 합니다.
- 페이지 로딩이 느려지지 않아야 합니다.

---

### 시나리오 4: 반응형 디자인 테스트

**목표**: 다양한 화면 크기에서 애플리케이션이 정상적으로 표시되는지 검증합니다.

**단계**:
1. 모바일 화면 (375px)에서 테스트
2. 태블릿 화면 (768px)에서 테스트
3. 데스크톱 화면 (1920px)에서 테스트

**확인 사항**:
- 레이아웃이 깨지지 않는지
- 버튼과 링크가 터치하기 쉬운 크기인지
- 텍스트가 읽기 쉬운 크기인지
- 이미지가 적절한 크기로 표시되는지

**예상 결과**: 모든 화면 크기에서 정상적으로 표시되어야 합니다.

---

### 시나리오 5: 접근성 테스트

**목표**: 모든 사용자가 애플리케이션을 사용할 수 있는지 검증합니다.

**확인 사항**:
1. 키보드 네비게이션
   - Tab 키로 모든 인터랙티브 요소에 접근 가능한지
   - Enter 키로 버튼 클릭 가능한지

2. 스크린 리더 호환성
   - 모든 이미지에 alt 텍스트가 있는지
   - 폼 필드에 라벨이 있는지
   - 버튼에 명확한 텍스트가 있는지

3. 색상 대비
   - 텍스트와 배경의 대비가 충분한지
   - 색상만으로 정보를 전달하지 않는지

**예상 결과**: 접근성 기준을 충족해야 합니다.
